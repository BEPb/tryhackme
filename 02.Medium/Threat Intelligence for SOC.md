## Задание 2
Обзор информации об угрозах

Для краткого обзора давайте повторим определение понятия «разведка об угрозах»,  которое обсуждалось в зале 
«Введение в разведку об угрозах в киберпространстве».

Анализ угроз — это анализ данных и информации с использованием инструментов и методов для создания значимых шаблонов 
для снижения потенциальных рисков, связанных с существующими или новыми угрозами, направленными на организации, 
отрасли, сектора или правительства.

Существуют различные классификации разведки угроз, основными типами которой являются:
- Стратегическая разведка:  разведывательная информация высокого уровня, которая анализирует ландшафт угроз 
  организации и выявляет области риска на основе тенденций, закономерностей и возникающих угроз, которые могут 
  повлиять на бизнес-решения.
- Техническая разведка: изучает доказательства и артефакты атак, которые использует противник. Группы реагирования 
  на инциденты могут использовать эту разведку для создания базовой поверхности атаки для анализа и разработки 
  механизмов защиты. 
- Тактическая разведка: Оценивает тактику, методы и процедуры (TTP) противников. Эта разведка может усилить контроль 
  безопасности и устранить уязвимости посредством расследований в реальном времени.
- Оперативная разведка: оценивает конкретные мотивы и намерения противника провести атаку. Группы безопасности могут 
  использовать эту разведку для понимания критических активов, доступных в организации (людей, процессов и 
  технологий), на которые могут нацеливаться субъекты угроз.
Эти классификации могут дать вам представление о том, как вы работаете с данными. Но в этой комнате мы 
  сосредоточимся только на технической разведке, использующей артефакты, созданные злоумышленниками, для улучшения 
  конвейера операций безопасности. Этот тип является наиболее распространенным из четырех классов и в основном 
  известен как разведка угроз на основе IOC.

Прежде чем применять данные разведки об угрозах к операциям по обеспечению безопасности, давайте сначала разберемся, 
как организации различаются по своим ролям в отношении данных разведки об угрозах.

Потребители и производители

Создаете ли вы базу знаний или потребляете знания других?

Общее понятие Threat Intelligence — это набор данных известных плохих IOC, собранных различными субъектами. Это 
могут быть вредоносные URL-адреса, на которых размещено вредоносное ПО, или IP-адреса подозрительных соединений. Но 
знаете ли вы, как эта информация собирается для утилизации аналитиками безопасности? Давайте сначала разграничим 
понятия производителей и потребителей Threat Intelligence.

Производители

Пример аналитика, работающего над отчетом в качестве производителя разведывательной информации об угрозах.
Производители разведданных об угрозах собирают, анализируют и распространяют данные разведданных об угрозах для 
других и для себя. Эти производители создают отчеты, рекомендации и ресурсы, которые распространяются в более 
широком сообществе кибербезопасности. В эту группу входят поставщики услуг кибербезопасности, исследовательские 
лаборатории и организации, специализирующиеся на сборе и интерпретации данных о возникающих киберугрозах.

Теперь производители обычно собирают данные, используя различные методы и приемы. Стандартные методы включают 
сетевой мониторинг, который включает мониторинг сетевого трафика организации для выявления аномального поведения 
изнутри или сервера honeypot, открытого снаружи.

Другим примером может быть набор IOC, основанный на внутренних инцидентах, обработанных организацией. Эти 
организации ожидают, что будет скомпрометировано большее количество инцидентов по сравнению с небольшими 
организациями с меньшим количеством активов или будет отслеживаться активность пользователей. Результаты этих 
наборов затем дополнительно анализируются, приписываются потенциальным субъектам угроз и в конечном итоге 
публикуются для помощи другим организациям.

Эти примеры резюмируют, что не каждая организация может быть производителем. Для этого требуется обширный набор 
собранных данных, способность определять ожидаемое нормальное поведение и способность анализировать и определять 
неизвестные потенциальные угрозы.

Потребители

С другой стороны, Потребители информации об угрозах — это организации или лица, которые потребляют информацию об 
угрозах, созданную Производителями. Информация, собранная из разных источников, используется для улучшения состояния 
безопасности организации.

Пример аналитика, являющегося потребителем разведывательной информации об угрозах.Каким образом эта группа обычно 
использует предоставленные ей разведывательные данные?

- Выявление уязвимостей - Потребители могут использовать опубликованные уязвимости, обнаруженные из-за нулевых дней, 
запущенных субъектами угроз, для выявления уязвимостей в инфраструктуре организации. Такие рекомендации, как 
  публикации CVE, обычно используются для определения того, затронута ли организация этим, и применения мер по 
  смягчению последствий при необходимости.
- Предотвращение и обнаружение. Потребители могут использовать IOC для предотвращения вторжений, блокируя эти 
  артефакты или обнаруживая их, применяя их к правилам обнаружения угроз.
- Реагирование на инциденты. Потребители могут использовать разведывательные данные для более эффективного 
  реагирования на инциденты, поскольку эти данные могут подтвердить вероятность атаки и потенциально приписываемые 
  ей тактику и методы.
- Сотрудничество с другими — обмен информацией предназначен не только для производителей, но и для потребителей. 
  Анализ данных IOC может по-прежнему требовать человеческой оценки, поэтому полезно обмениваться информацией, 
  которая проверена и полезна для операций по обеспечению безопасности.

## Задание 3
Ваша организация определила, что вы являетесь потребителем информации об угрозах из надежных источников; ваша задача 
— применять концепции потребителя, развертывая элементы управления для предотвращения угроз в вашей инфраструктуре.

Используя наши текущие знания об угрозах, мы будем использовать IOC из надежных источников для развертывания средств 
контроля безопасности, которые предотвратят вредоносную активность в нашей инфраструктуре.

Для начала мы можем упростить типы IOC, которые обычно выделяются в потоках данных об угрозах:

- Домены — обычно приписываются URL-адресам, используемым для размещения вредоносных файлов, обратных вызовов C2 или 
доменам электронной почты, используемым для рассылки спама.
- IP-адреса — обычно приписываются адресам, известным тем, что с них осуществляются атаки с внешних ресурсов или 
  исходящие обратные вызовы от вредоносного ПО.

#### Блокировка IP через брандмауэр

Блокировка IP-адресов — это известная мера безопасности, которая включает блокировку входящего или исходящего 
сетевого трафика на основе IP-адреса устройства, пытающегося инициировать сетевое соединение. Обычно это делается с 
помощью брандмауэра — системы безопасности, которая контролирует трафик на основе предопределенных правил.

Настройка правил брандмауэра может оказаться сложной задачей, но наличие указания запретить подключения с известного 
вредоносного IP-адреса станет хорошим началом для предотвращения вредоносных подключений, таких как:
- Предотвращайте несанкционированные подключения к внешним приложениям, которые могут повлиять на бесперебойную 
  работу сервиса или поставить его под угрозу из-за известной уязвимости.
- Предотвращать попытки подключения к инфраструктуре злоумышленника после успешного выполнения вредоносного ПО.

Схема сети блокировки IP.

#### Блокировка домена через шлюзы электронной почты

Подобно блокировке IP-адресов, мы можем настроить шлюзы электронной почты, чтобы запретить известным вредоносным 
доменам пересылать входящие сообщения электронной почты на основе домена отправителя. Шлюзы электронной почты также 
зависят от набора правил, который должен содержать список заблокированных доменов, известных отправкой спама или 
фишинговых писем. После заполнения списка заблокированных сообщений шлюз электронной почты не позволяет субъектам 
угроз достигать почтовых ящиков целевых пользователей в организации.

Схема сети блокировки домена шлюза электронной почты.

Предотвращение попадания спама в почтовые ящики сотрудников уменьшает потенциальные поверхности атак субъекта угрозы,
скомпрометировавшего организацию. В большинстве случаев захват организации начинается с выполнения вредоносного 
вложения или отправки учетных данных на фишинговый веб-сайт. Дополнительный уровень предотвращения немного снижает 
нагрузку на возможности пользователей по распознаванию фишинга.

#### Блокировка домена через DNS Sinkhole

DNS Sinkhole — это мера безопасности, которая смягчает соединения с вредоносным доменом. Обычно это делается путем 
перенаправления всех DNS-запросов с известного вредоносного домена на sinkhole, предотвращая разрешение на их 
IP-адреса.

Диаграмма сети DNS Sinkhole.

Охота на провалившиеся домены

Используя журналы, созданные SIEM, вам поручено охотиться за всеми доменами, которые ваш DNS Sinkhole 
идентифицировал как вредоносные. Команда Threat Intelligence предоставила один домен для начала вашей активности:

Кроме того, для завершения расследования вам могут также понадобиться следующие шаблоны запросов KQL :
```commandline

```
Обратите внимание, что вам необходимо удалить defang из вашего поискового запроса.

Наконец, события сохраняются в filebeat-* индексе. Убедитесь, что поисковый запрос находится между 14.02.2023 и 17.02.2023. Удачи!

#### Оптимизация возможностей обнаружения

Реализация обнаружения на основе IOC может быть довольно простой, поскольку можно подумать, что мы можем развернуть 
правило списка блокировки для известных вредоносных IOC. Пример набора вариантов использования обнаружения приведен 
ниже для каждого распространенного Threat Intel IOC.


- **IP-адрес**	
  Подключения через журналы брандмауэра, в которых направление подключения указывает на потенциальную первопричину:
  1. Исходящее соединение с вредоносным IP-адресом указывает на потенциальное выполнение вредоносного ПО, таким 
  образом 
    осуществляя связь с IP-адресом злоумышленника.
  2. Входящее соединение с вредоносного IP-адреса свидетельствует о попытке вторжения со стороны злоумышленников, 
      демонстрируя следы фазы, предшествующей эксплуатации.

- **URL**	
Подключения через прокси-сервер регистрируются, при этом метод HTTP определяет характер подключения:
  1. HTTP- запросы GET указывают на потенциальную загрузку вредоносных файлов или доступ к фишинговому сайту.
  2. Более того, запросы HTTP POST указывают на потенциальную передачу учетных данных или утечку украденных файлов.

- **Домен**	
Вредоносные домены, обнаруженные в журналах DNS, напрямую указывают на вредоносную активность в одном из следующих 
  случаев:
  1. На домене размещено вредоносное ПО или дополнительные файлы для его цепочки выполнения.
  2. Домен является фишинговым сайтом.
  3. Домен используется для соединения C2.


Приведенные выше сценарии описывают использование общедоступных IOC для поражения любых подозрительных соединений в 
различных источниках данных, таких как брандмауэры, DNS и прокси-серверы. Однако такой тип настройки может 
потребовать постоянной тонкой настройки правил для удовлетворения роста IOC. 

Мы можем объединить некоторые методы профилактики, рассмотренные в Задаче 3, для обнаружения подозрительного трафика.


- **DNS- воронка**	Домены, разрешающие обратную связь (127.0.0.1 или 0.0.0.0), могут указывать на соединение с 
вредоносным доменом на основе конфигурации черного списка DNS-каналов.

- **Блокировка IP-адресов брандмауэра** Заблокированные соединения с определенным IP-адресом могут указывать на 
  вредоносную активность и заслуживают расследования. Журналы, сгенерированные при блокировке IOC, дают больше 
  контекста о соединении.

- **Блокировка прокси**	Заблокированные веб-соединения могут указывать на злонамеренную попытку доступа к вредоносному 
ПО или фишинговому сайту. Служба Proxy могла бы предоставить больше информации, если бы имела возможности 
  тегирования для отражения вредоносных соединений с помощью тегов.
- **Блокировка почтового шлюза**	Письма, заблокированные на основании домена отправителя, могут указывать на попытку 
  рассылки спама злоумышленником.

При этом правила обнаружения тонкой настройки полагаются только на обновления списка блокировки из тактик 
предотвращения. Таким образом, этот способ представляет оптимальный способ предотвращения и обнаружения вредоносной 
активности на основе Threat Intelligence IOC.

#### Пересмотр правил Sigma

Как обсуждалось в модуле Detection Engineering, Sigma — это открытый исходный код общего языка сигнатур для 
описания событий журнала в структурированном формате. Это позволяет аналитикам безопасности быстро обмениваться 
методами обнаружения.

В этой задаче мы будем использовать следующее правило сигмы для поиска проваленных доменов. 
```commandline
пользователь@threatintel:~/
title: DNS Sinkhole
author: TryHackMe User
description: Sigma rule for sinkholed DNS queries 
logsource:
 category: dns
detection:
 select_sinkholed:
   dns.resolved_ip:
     - '0.0.0.0'
 condition: select_sinkholed
falsepositives:
 - Unknown
status: experimental
level: medium
tags:
 - dns
 - filebeat
```
Правило Sigma выше ищет DNS- запросы, разрешающие 0.0.0.0. Как обсуждалось ранее, такие случаи могут указывать на 
соединение с известным подозрительным доменом на основе конфигурации DNS Sinkhole.

Играем с ElastAlert и Uncoder.io

Для эмуляции обнаружения образца мы будем использовать Uncoder.io для перевода ранее упомянутого правила Sigma в 
ElastAlert.  Убедитесь, что преобразование установлено из Sigma в ElastAlert, прежде чем нажать Translate.

Конвертация Sigma в ElastAlert с использованием Uncoder.io.

Прежде чем использовать сгенерированное правило, давайте кратко рассмотрим ElastAlert.

ElastAlert — это фреймворк с открытым исходным кодом для оповещения об аномалиях, пиках или других интересных 
закономерностях, обнаруженных в данных, хранящихся в Elasticsearch. Он интегрируется с Elasticsearch, Kibana и 
другими инструментами в экосистеме Elasticsearch и может быть настроен для отправки оповещений в различные внешние 
службы, такие как Email, Slack, PagerDuty и другие.

Полученное правило ElastAlert от Uncoder.io содержит следующую информацию:

- тревога	- Тип оповещателя для использования. Значение  debug будет регистрировать информацию об оповещении на 
уровне информации.
- фильтр -	Список фильтров запросов Elasticsearch. Текущий запрос ищет все домены, разрешающиеся в 0.0.0.0 .
- индекс - Имя индекса, по которому будет производиться поиск. В нашем текущем контексте правило будет сканировать 
  содержимое индекса  filebeat-* .
- реалерт - Эта опция позволяет игнорировать повторяющиеся оповещения в течение некоторого времени. Значение 
  minutes: 0 будет генерировать все оповещения, несмотря на свою избыточность. 
- тип - Тип правила для использования. Значение any будет генерировать оповещение при каждом успешном возврате запроса.

Теперь, когда мы внедрили ElastAlert, войдите в машину через SSH , используя предоставленные учетные данные в Задаче 
2 (user:tryhackme) , и перейдите в  каталог ~/elastalert . Вы можете увидеть, что каталог содержит файл конфигурации 
и подкаталог.

```commandline
пользователь@threatintel:~/elastalert
user@threatintel:~/elastalert$ ls
config.yaml  rules
user@threatintel:~/elastalert$ ls rules/
sinkhole.yaml
```
Файл config.yaml содержит все конфигурации, необходимые для подключения и запроса к нашему экземпляру Elasticsearch ,
в то время как каталог ~/elastalert/rules содержит правило-заполнитель. Вы можете заполнить это правило переведенным 
правилом Sigma в ElastAlert из Uncoder.io.

Примечание:  После копирования правила Elastalert, сгенерированного Uncoder.io, выполните следующее, чтобы очистить 
синтаксис и указать ElastAlert на правильный индекс.

В поле описания удалите строку, начиная с  Author:и до конца строки. Это удаляется, чтобы сделать синтаксис правила 
ElastAlert действительным.
В поле индекса замените winlogbeat-*на filebeat-*.
После выполнения указанных выше шагов правило ElastAlert должно быть похоже на приведенное ниже.

```commandline
пользователь@threatintel:~/elastalert/rules
user@threatintel:~/elastalert/rules$ cat sinkhole.yaml
alert:
- debug
description: Sigma rule for sinkholed DNS queries.
filter:
- query_string:
    query: dns.resolved_ip:"0.0.0.0"
index: filebeat-*
name: dns_sinkhole
priority: 3
realert:
  minutes: 0
type: any
```
После настройки правила sinkhole.yaml вернитесь в каталог elastalert и запустите ElastAlert.

```commandline
пользователь@threatintel:~/elastalert
user@threatintel:~/elastalert/rules$ cd ~/elastalert
user@threatintel:~/elastalert$ elastalert --start 2023-02-16T00:00:00 --verbose 2>&1 | tee output.txt
```
Приведенную выше команду elastalert можно разбить следующим образом:

ElastAlert выполняет настроенное нами правило с 16.02.2023 по настоящее время.
Он также обеспечивает подробный вывод. 
Наконец, фрагмент использует  2>&1 | tee output.txt для записи результатов в output.txt . Обратите внимание, что были использованы дескрипторы файлов, поскольку вывод записывается на уровне INFO .
После выполнения команды вам может потребоваться подождать несколько секунд, чтобы завершить начальный запуск. Следующая строка указывает на то, что выполнение завершено: X query hits (X already seen), X matches, X alerts sent

Увидев строку выше, вы теперь можете остановить выполнение команды elastalert с помощью CTRL + C.
